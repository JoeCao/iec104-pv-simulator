# IEC104 连接稳定性问题复盘与建议

## 背景

- 现象：模拟器运行约 2 小时后，平台侧看到数据不再更新。
- 现场：服务端模拟器进程仍在运行，日志仍持续打印状态行。
- 版本：2026-04-23 晚部署版本（后续切换到 2000 点位配置）。

## 现象与关键观察

### 1) 模拟器未停止，数据仍在计算与上送

服务端日志持续出现如下信息，说明模拟器在正常运行并持续发送 COT=3：

- `总点数=2000 读点=1934 写点=66 | COT=3(10s)=xxxx | COT=3(total)=yyyy`

这说明“数据不更新”并非模拟器主循环停止。

### 2) 存在大量连接/断开事件

日志中有大量如下连接事件：

- `客户端连接请求: <ip>`
- `客户端已连接`
- `连接已激活 (STARTDT)`
- `客户端已断开`

并且除采集端 IP 外，存在多个外部 IP 访问 2404 端口。

### 3) 你的 zdaq IP 也频繁重连

对 zdaq IP（如 `117.89.43.132`）的日志统计显示连接请求次数很多，且经常出现连接后很快断开并再次重连。

## 根因分析

核心根因是 **IEC104 服务端模式 + 公网暴露扫描流量** 共同导致：

1. 当前服务端默认使用 `CS104_MODE_SINGLE_REDUNDANCY_GROUP`。  
   在该模式下，新激活连接可能导致旧连接被 `deactivate`。
2. 2404 端口暴露在公网，扫描器/未知客户端持续连接，会抢占或扰动 active 会话。
3. 结果是：
   - TCP 可能仍是 ESTAB（看起来“连着”）；
   - 但 IEC104 应用层会话可能已非 active，zdaq 侧表现为“在线但无新数据”；
   - 触发 zdaq 重连后又恢复，随后可能再次被扰动。

## 为什么“有连接却没数据”？

IEC104 分层上：

- TCP ESTAB 只表示传输层连接存在；
- 还需要应用层状态正常（例如 STARTDT 激活、总召/变化上送链路持续有效）。

所以“连接存在”不等于“正在稳定收数”。

## 结论

- 这次问题不是“模拟器停止模拟”，而是“连接会话被外部连接扰动导致采集侧失活或重连”。
- 本质属于安全边界和连接模式问题，而不是单纯点位规模问题。

## 建议与优先级

## P0（立即执行）

1. **防火墙白名单（首选）**  
   只允许采集端源 IP 访问 2404；拒绝其余来源。
2. **持续观察连接事件**  
   关注日志中 `[连接]` 是否显著下降，zdaq 是否稳定保持 active。

## P1（增强稳定性）

1. **协议层白名单（应用层兜底）**  
   在 `connectionRequestHandler` 增加允许 IP 列表过滤。
2. **考虑调整 server mode**  
   评估切换为 `CS104_MODE_CONNECTION_IS_REDUNDANCY_GROUP`，避免连接互踢。  
   注意：即便切换模式，仍建议保留防火墙白名单。

## P2（中长期）

1. 如果采集端出口 IP 会变化，建议：
   - 固定公网出口 IP，或
   - 通过 VPN/专线接入，减少公网暴露面。
2. 评估 IEC 62351/TLS 能力（若采集端与平台支持）。

## 运维建议（可直接执行）

1. 启动时显式使用 2000 点配置（避免回到 10000 默认文件）：

```bash
pv_ctl restart /opt/pv_simulator/config/sim_rules_2000.csv 2404
```

2. 观察状态与日志：

```bash
pv_ctl status
pv_ctl log
```

3. 关注以下关键信号：

- `已加载点位: 2000 (读=1934, 写=66)`
- `COT=3(10s)` 持续变化
- `[连接]` 日志是否仍出现大量未知 IP

## 备注

- IEC 60870-5-104 标准明文 TCP 通常不提供“用户名/密码登录”机制，安全主要依赖网络边界控制。
- 因此，将 2404 直接暴露公网风险较高，白名单和网络隔离是必要措施。
